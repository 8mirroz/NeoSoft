# ADR 001: Технологический стек Combo Fever Engine

## Статус
Принято (Active)

## Контекст
Combo Fever Engine требует высокой производительности при расчете геометрии сетки, отзывчивого ввода без задержек, а также процедурных визуальных эффектов, способных масштабироваться под мобильные устройства. Необходима строгая типизация и архитектурное разделение логики и визуального представления (Controller-First Design / Four Games pattern).

## Решение
Для реализации Combo Fever Engine выбран следующий технологический стек:

1. **Базовая платформа**:
   - **Godot 4.3+**: Использование последней стабильной версии Godot Engine.
   - **GDScript 2.0**: Статически типизированный скриптовый язык с поддержкой сигналов, лямбда-выражений и статических классов для математического ядра.

2. **Архитектурный паттерн**:
   - **Controller-First**: Строгое разделение на логический контроллер поля (`BoardController`), логическую модель (`BoardModel` / `LogicState`) и визуальное представление (`BoardView` / `GemView`). Логика не зависит от `Node` и `SceneTree`, что позволяет проводить мгновенные тесты и headless-симуляции.

3. **Аудиовизуальные технологии**:
   - **Процедурная отрисовка CanvasItem `_draw()`**: Используется для отрисовки интерфейсов, светящихся контуров Combo Window и спец-сфер.
   - **GPUParticles2D / CPUParticles2D**: Системы частиц для визуализации взрывов, полета наводящихся сфер и Fever-эффектов. CPUParticles2D используется как автоматический fallback на слабых мобильных устройствах для экономии GPU.
   - **AudioServer & AudioEffect**: Управление динамическими аудиослоями Fever Mode через стандартные шины микшера Godot.

4. **Инструменты валидации и тестов**:
   - **Gut (Godot Unit Testing)**: Использование фреймворка юнит-тестирования для валидации всех 16 тест-кейсов (формулы комбо, очередность ввода, распознавание фигур, спец-комбинации).

## Последствия
- Высокая скорость выполнения расчетов (менее 1мс на шаг симуляции).
- Полная тестируемость ядра без запуска графического интерфейса.
- Легкая портируемость математической логики.
- Требуется строгая дисциплина ведения состояний ячеек (`CellState`) для исключения рассинхронизации логики и визуала.
