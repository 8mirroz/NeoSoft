# System Design — Combo Fever Engine

Этот документ предоставляет детальный технический дизайн **Combo Fever Engine (CFE)** для версии 4.0. В нём зафиксированы формулы, структуры данных, алгоритмы и переходы состояний ячеек игрового поля.

---

## 1. Математический баланс и формулы

Главная задача формулы ComboPower — не допустить экспоненциального взрыва очков и эффектов, удерживая игру в сбалансированных рамках сложности.

### 1.1 Формула расчёта RawComboPower
Каждый матч или активация спец-сферы рассчитывает исходную силу:

$$RawComboPower = BaseMatchValue + (ChainIndex \times ChainMultiplier) + ShapeWeight + SpecialWeight + SpeedBonus + CascadeBonus + ObjectiveBonus$$

Где:
- $BaseMatchValue$: Базовая ценность матча (обычно 100).
- $ChainIndex$: Порядковый номер комбо в цепочке (x1, x2, x3...).
- $ChainMultiplier$: Множитель за длину комбо (например, 25).
- $ShapeWeight$: Бонус за форму совпадения (Line-3 = 0, Line-4 = 50, Square-2x2 = 60, L/T = 75, Line-5 = 100).
- $SpecialWeight$: Ценность активированных спец-сфер (Луч = 50, Взрыв = 70, Наведение = 65, Призма = 120).
- $SpeedBonus$: Бонус за скорость совершения свайпа с момента стабилизации поля (рассчитывается по кривой затухания, макс +40).
- $CascadeBonus$: Множитель за автоматический матч при осыпании (+15 за каждый каскад).
- $ObjectiveBonus$: Бонус, если матч уничтожил целевой элемент уровня (+30).

### 1.2 Нормализация ComboPower
Для удержания силы эффектов в рамках баланса применяется нелинейная сигмоидная кривая (Difficulty Curve) с последующим clamp:

$$ComboPower = clamp(f(RawComboPower, DifficultyProfile), MinPower, MaxPower)$$

Где:
- $MinPower = 100$
- $MaxPower = 1500$
- Функция $f$ сжимает сверхвысокие значения очков, предотвращая автоматическое прохождение уровня при Fever Mode.

## 2. Диаграмма переходов состояний ячеек (Cell State Matrix)

Каждая ячейка логического поля в `BoardStateEngine` подчиняется строгой матрице переходов. Свайп игрока разрешается только на ячейках в состоянии `STABLE`.

```text
  [BLOCKED] ──(Инициализация уровня)──> [STABLE] <──(Стабилизация каскада)── [FALLING]
                                          │                                     ▲
                                (Свайп игрока / Резерв)                         │
                                          │                                (Осыпание вниз)
                                          ▼                                     │
     [RESERVED] ──(Валидация)──> [RESOLVING] ──(Матч)──> [LOCKED] ──────────────┘
```

### Таблица валидности переходов состояний

| Исходное состояние | Целевое состояние | Условие перехода | Валидность |
|---|---|---|---|
| **STABLE** | **RESERVED** | Свайп игрока добавлен в буфер ввода | **Валидно** |
| **STABLE** | **RESOLVING** | Прямой ход игрока / проверка матча | **Валидно** |
| **FALLING** | **STABLE** | Сфера приземлилась в ячейку | **Валидно** |
| **RESOLVING** | **LOCKED** | Сфера совпала, запускается взрыв | **Валидно** |
| **LOCKED** | **FALLING** | Взрыв завершён, ячейка пуста, сферы сверху начинают осыпаться | **Валидно** |
| **FALLING** | **RESOLVING** | Сфера падает, но свайп запрещён | *Невалидно* (Ход блокируется) |
| **RESERVED** | **STABLE** | Очередь ввода отменена / истекло время | **Валидно** |

## 3. Детальные алгоритмы подсистем

### 3.1 Алгоритм буферизации ввода (`InputBufferController`)
При каждом свайпе во время каскада:
```text
1. Получить координаты from_cell и to_cell.
2. Проверить CellState обеих ячеек.
3. Если (from_cell.state == STABLE и to_cell.state == STABLE):
      Выполнить swap немедленно.
4. Иначе если (размер очереди < max_queue_size):
      cell_id_from = grid.get_gem_id(from_cell)
      cell_id_to = grid.get_gem_id(to_cell)
      
      Создать QueuedMove {
         from: from_cell,
         to: to_cell,
         expected_from_id: cell_id_from,
         expected_to_id: cell_id_to,
         expires_at: Time.get_ticks_msec() + (buffer_lifetime * 1000)
      }
      Перевести ячейки в RESERVED.
      Добавить в очередь.
      Запустить таймер ожидания стабилизации.
5. Иначе:
      Отклонить ввод, проиграть тихий dull-pop звук.
```

### 3.2 Алгоритм Target Priority System
Когда наводящаяся сфера взрывается, она запрашивает цель:
```text
1. Собрать все активные цели уровня (Level Objectives) на поле.
2. Отфильтровать цели, находящиеся под препятствиями (блокированы льдом, цепями).
3. Если есть заблокированные цели:
      Выбрать цели в углах поля (координаты 0 или max_index).
      Если угловых нет, выбрать цель с наименьшим HP препятствия.
      Иначе выбрать любую заблокированную цель.
4. Если заблокированных целей нет:
      Выбрать обычные цели уровня.
5. Если целей уровня на поле нет (все собраны, остался только счет):
      Выбрать препятствия с низким HP (для зачистки поля).
      Если препятствий нет, выбрать случайную сферу редкого цвета.
6. Вернуть Vector2i выбранной ячейки.
```

## 4. Метрики сложности и автобаланс уровня

Combo Fever Engine логирует показатели для headless-симулятора MCTS (Monte Carlo Tree Search), описанного в пользовательском навыке `godot-mobile-match3`:

- **Win-Rate Калибровка**:
  Для легких уровней (1-3) время Combo Window увеличивается до 2.0 сек, а Fever-порог снижается. Headless-симулятор должен показывать win-rate $\ge 95\%$.
  Для сложных уровней (8-10) Combo Window сжимается до 1.1 сек, Fever-порог повышается. Симулятор должен показывать win-rate в диапазоне $20\% - 35\%$.
- **Idle Time Reduction (Сокращение ожидания)**:
  CFE стремится свести метрику `player_idle_time_during_cascade` к минимуму. Целевой показатель MVP: общее время блокировки ввода за уровень должно снизиться на $60\%$ по сравнению с классической пошаговой моделью v2.
