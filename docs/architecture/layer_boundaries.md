# Архитектурные слои и системные границы — Genesis v5.1

Для обеспечения спортивной честности, легкой отладки и высокой производительности на мобильных устройствах проект **Neo Soft Frost** разделен на **6 изолированных слоев**.

```text
┌─────────────────────────────────────────────────────────┐
│                Layer 6: Telemetry + Auto-Balance         │
│   Записывает метрики баланса, replay-файлы, MCTS тесты   │
├─────────────────────────────────────────────────────────┤
│                Layer 5: Juicy Feedback Director           │
│   Оркестрирует частицы, аудиослои, haptics, тряску    │
├─────────────────────────────────────────────────────────┤
│              Layer 4: Special Sphere Ecosystem            │
│   Генерирует спец-сферы по правилам, считает приоритеты │
├─────────────────────────────────────────────────────────┤
│              Layer 3: Shape Economy                       │
│   Детектирует 11 форм совпадений, разрешает конфликты   │
├─────────────────────────────────────────────────────────┤
│            Layer 2: Controlled Probability                │
│   Считает Assisted Drop, контролирует глубину и Fever  │
├─────────────────────────────────────────────────────────┤
│              Layer 1: Deterministic Core                  │
│    BoardLogic, GravitySolver, FSM- ResolvePipeline      │
└─────────────────────────────────────────────────────────┘
```

---

## 1. Спецификация слоев

### Layer 1: Deterministic Core (Детерминированное ядро)
* **Назначение**: Математически точная симуляция сетки 8x8. Это чистый автомат состояний, не зависящий от графического вывода.
* **Ключевые модули**: `BoardLogic`, `GravitySolver`, `ResolvePipeline`, `InputBufferController`.
* **Правило взаимодействия**: Запрещено ссылаться на любые внешние шейдеры, UI или VFX-менеджеры. При любых изменениях состояния вызываются сигналы в `EventBus`.

### Layer 2: Controlled Probability (Управляемая вероятность каскадов)
* **Назначение**: Расчет весов выпадения гемов для удержания игрока в состоянии "pleasure flow" без ощущения фальши.
* **Ключевые модули**: `ControlledCascadeEngine`, `CascadeGovernor`, `BalanceGovernor`, `DropRngController`, `ComboFeverController`.
* **Правило взаимодействия**: Читает детерминированные сиды и JSON-конфиги. Может накладывать вето (Color Interlocking) на любые совпадения Layer 1 для гашения бесконечных каскадов.

### Layer 3: Shape Economy (Распознавание форм)
* **Назначение**: Сканирование сетки для нахождения 11 типов сложных геометрических фигур и классификации их ценности.
* **Ключевые модули**: `ShapeDetector`, `ShapeClassifier`.
* **Правило взаимодействия**: Вызывается из `ResolvePipeline` сразу после нахождения базовых совпадений. Передает результат фабрикам через типизированный контракт.

### Layer 4: Special Sphere Ecosystem (Экосистема спец-сфер)
* **Назначение**: Создание и активация 10 типов спец-сфер, 15 комбо-сочетаний и умный выбор приоритетных целей.
* **Ключевые модули**: `SpecialSphereFactory`, `SpecialComboResolver`, `TargetPrioritySystem`, `SpecialBalanceGuard`.
* **Правило взаимодействия**: Применяет эффекты к доске Layer 1 строго через транзакции `ResolvePipeline`. Считает приоритеты по формуле весов.

### Layer 5: Juicy Feedback Director (Режиссер сочности)
* **Назначение**: Перевод геймплейных событий в визуально-звуковой восторг (VFX/SFX).
* **Ключевые модули**: `FeedbackDirector`, звуковые и тактильные роутеры, gem анимации.
* **Правило взаимодействия**: Пассивно слушает `EventBus`. Не имеет права мутировать логическое состояние доски, менять цвета гемов или влиять на механику. Поддерживает `SAFE`-режим для старых Android-устройств.

### Layer 6: Telemetry + Auto-Balance (Телеметрия и автобаланс)
* **Назначение**: Запись логов сессий, replay-файлов, проведение headless-симуляций (MCTS) для проверки сходимости уровней.
* **Ключевые модули**: `BalanceTelemetryLayer`, `ReplayRecorder`, `ReplayPlayer`, MCTS симулятор.
* **Правило взаимодействия**: Слушает `EventBus` и сохраняет данные в локальные JSON-файлы. Ничего не мутирует в процессе игры.

## 2. Разрыв жестких связей (Decoupling)

Все взаимодействие между логическим ядром (Layers 1-4) и представлением/аналитикой (Layers 5-6) осуществляется строго через **EventBus** с передачей типизированных неизменяемых контрактов данных. Ни один скрипт из `scripts/core_match3/` не должен содержать слов `Node2D`, `ParticleProcessMaterial`, `AudioStreamPlayer2D`, `Viewport` или `HUD`.
